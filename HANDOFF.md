## The 8 modes

`ask` (research-only, never mutates) · `env` (agent-environment/CLAUDE.md setup) · `define` (spec) · `design` (design.md + decisions.md, uses vibe-architect) · `plan` (plan/checklist/verification-plan, uses vibe-test-designer) · `build` (**only mutating mode**, in-place on `latest`, checkpoint loop, human approval gate) · `verify` (runs verification plan) · `review` (independent quality pass). Pipeline: `define → design → plan → build → verify → review`; `ask`/`env` usable anytime.

Key invariants: read-only modes write only inside their run dir; `build` requires explicit written action-specific approval (vague "ok" rejected); run dirs under `.claude/vibe-reports/<UTC>/` with a plain-text `latest` pointer (not symlink — Windows).

## Three feedback-driven refinements already applied

1. **Greenfield detection by source-tree content**, not just repo markers — bare greenfield vs. *scaffolded greenfield* (repo generated by a scaffolder like the user's `generate-project`, where `src/` holds only placeholders) vs. existing codebase. Scaffolded repos are treated as greenfield for the interview but their generated tooling is honored as constraints. (`SKILL.md`, `modes.md`)
2. **Config-based tool detection** — a tool counts as "in use" only if configured (`pyproject.toml [tool.*]`, dedicated config files, dev deps, Makefile targets), never merely on PATH. flake8 and pylint added alongside mypy. (`python-stack.md`)
3. **v2 features**: (a) optional **phase grouping** in plans; (b) **per-agent model config** via `.claude/vibe-coding.local.md` frontmatter (`models:` map + `auto_max_checkpoints`), read by `probe_subagents.py`; (c) **autopilot `--auto`** — `vibe-overseer` assumes the approver role; the user's invocation is the coarse-grained written grant; overseer is *required* (no inline fallback for the approver role, ever); hard-stop on reject/blocker/out-of-repo mutation; audit trail in `state.json.build.auto` + `build-log.md`.

## Description optimization (completed, with caveats worth knowing)

The skill-creator description-optimizer (`scripts/run_loop.py`) was run against 22 trigger queries (`skills/vibe-coding-workspace/trigger-eval.json`, user-reviewed + 2 autopilot queries). Three obstacles were worked around; **the plugin cache was never modified** — all patches live in a scratchpad copy at `/private/tmp/claude-501/.../scratchpad/skill-creator-patched/`:

1. **Detection bug in the optimizer**: it credited a trigger only if a probe's *own* hashed clone was invoked; with 10 identical parallel clones this undercounted real triggers. Patched to credit any same-description clone.
2. **Generation half (`improve_description` → `claude -p`) is unreliable in this nested environment** — fails persistently even with retry. So automated candidate generation does not work here; candidates must be hand-written and scored with the (working) measurement half.
3. **`ProcessPoolExecutor` + macOS `spawn`** re-imports loose driver scripts as `__main__` → worker crashes. Any custom driver calling `run_eval` **must** guard with `if __name__ == "__main__":`.

Clean measured result: the **pre-optimization** description already scored 88% held-out test accuracy, **100% precision** (zero false triggers on 10 near-miss should-not queries), 75% recall. The only genuine miss was autonomous/autopilot phrasing. A hand-written candidate adding autopilot language was verified head-to-head: the autonomous-build query went 0/3 → **3/3**, with **zero new false triggers**. That candidate is now the live `SKILL.md` description. (The remaining `/vibe …` literal "misses" are test-rig artifacts — the rig registers the skill under a hashed command name, not `/vibe`, so `/vibe` triggers correctly only once actually installed.)
